Background Info: Command Line & Git

Command Line

Usually in the course of building a project, we make use of the command line to run scripts and programs. Programmers often build tools that don't have a GUI (as we are doing now), and in those cases the terminal is the only place we can use the tools we need. We won't be doing anything too complex, but there are a few common commands used in Unix shells (such as Git bash) that are good to know.

Commands usually follow the following format: [command name] [options/flags] [arguments]. The command name is self-explanatory, but what about the other two?

Options or flags are arguments we provide to make the command behave a bit differently. As their name implies, they're optional, so most of the time you won't have to bother about writing anything. However, I've included some information for your general knowledge. Options are prefixed with one or two dashes (- or --). Here's an example, using the ls command, which lists all files in the current directory:


$ ls
example_folder file1.txt      file2.txt
$ ls -l	
total 16
drwxr-xr-x  2 daniellok  staff  64 Jun 14 14:23 example_folder
-rw-r--r--  1 daniellok  staff  41 Jun 14 14:23 file1.txt
-rw-r--r--  1 daniellok  staff  12 Jun 14 14:22 file2.txt

As you can see, the addition of the -l option makes the command not only list all the files, but also their metadata (e.g. size, permissions, creation date).

Arguments are anything else that the command needs—for example, when executing the command to delete a file, you need to specify the file's name as an argument. This is pretty much the same as an argument to a function in typical programming jargon.

Here is a list of commands that you'll probably find yourself using a lot:

  • ls: lists the files in the current directory
  • cd [directory name]: change directory to the one specified
  • mkdir [directory name]: make a directory with the specified name
  • rm [file name]: removes the specified file (add the -r option to delete folders)
  • cat [file name]: display the contents of a file (useful if you want to check a file without having to open it in your editor)

Git

As mentioned in the Requirements page, Git will be our source control tool. I'd recommend looking at a guide, but honestly, even though I did a couple of Git tutorials, it still took me about a year to be comfortable enough that I don't have to google every command. Don't hesitate to ask questions if you're not sure how to use it, I get that it can be really confusing if you've never used it before.

That being said, here are the steps to start our project:

  1. cd into the folder you want to start this project in.
  2. Enter git clone https://github.com/daniellok/tda-api-walkthrough.git. This copies our tda-api-walkthrough repository from GitHub onto your computer.
  3. cd tda-api-walkthrough and ls. You should only see a file called readme.md. All the code will be provided by you ðŸ˜Š.
We'll be creating a lot of files throughout this project, and we'll want to upload them to GitHub. Official changes to a repo (e.g. additions, deletion), are called commits. To make commits to our repo, you'll need to do a few things:
  1. git add [filename]. This adds the specified file to a list of stuff to be committed (you can add multiple files in one commit).
  2. git commit -m "[message]". This commits all added files, and adds a commit message (typically these describe what changes you've made).
  3. git push origin [branch name]. This uploads your commit to GitHub, so I'll be able to see it as well. The branch we'll use is master, but note that this is considered bad practice—the master branch is supposed to be reserved for peer-reviewed code which is stable. Individual changes should be done on your own branch. However, this is a small project so it's fine.
I'd expect there to be some hiccups along the way, but no worries! It's all part of the learning process.
